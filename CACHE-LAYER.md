# Cache Layer Architecture

Redis is a first-class component in MWECAU DigiVote. It serves multiple critical roles simultaneously.

## Redis Instance Layout

- **DB 0**: Celery broker + result backend
- **DB 1**: Django cache (default) + sessions

All Django cache keys are prefixed with `digivote:`.

## What Lives in Cache

### 1. Django Sessions
```python
SESSION_ENGINE = 'django.contrib.sessions.backends.cache'
SESSION_CACHE_ALIAS = 'default'
```
Fast session access and easy horizontal scaling.

### 2. Brute-force Protection
- `login_attempts:{registration_number}`
- `login_lockout:{registration_number}`
- TTL: 15 minutes on lockout

### 3. Password Recovery State
- OTP codes
- Security question sets (temporary)
- "Questions passed" flags
- Rate limiting counters

### 4. Election Monitoring (short-lived)
- Turnout stats (`election_turnout:{election_id}`) — 10 second TTL
- Used by observers and commissioners only

### 5. Election Results
- `election_results:{election_id}`
- TTL: 30 seconds while election is active, 1 hour after it ends
- Generated by `election/services.py:build_election_results()`
- Invalidated on vote submission and when results are published

## Caching Code Patterns

### Short-lived polling data
```python
data = cache.get(cache_key)
if data is None:
    ... compute ...
    cache.set(cache_key, data, 10)
```

### Results (with smart TTL)
```python
timeout = 3600 if election.has_ended else 30
cache.set(cache_key, results, timeout)
```

### Atomic login protection
```python
if cache.get(lockout_key):
    ...
attempts = cache.get(attempts_key, 0) + 1
cache.set(attempts_key, attempts, timeout=900)
if attempts >= 5:
    cache.set(lockout_key, True, timeout=900)
```

## Why Redis (not just local memory)

- Sessions must survive worker restarts and scale across processes
- Shared state for rate limiting across all requests
- Celery needs a broker anyway — using the same Redis avoids another service
- Future multi-instance architecture (multiple web servers) requires shared cache

## Cache Invalidation Strategy

- **Time-based** for most things (preferred)
- **Explicit delete** only on vote submission for results and on successful password reset flows
- No complex cache tagging — keep it simple and predictable

## Monitoring & Tuning

- `TIMEOUT: 300` default
- `max_connections` tuned in both django-redis and Celery transport options
- Connection pool shared conceptually between Celery and cache (different DB indexes)
